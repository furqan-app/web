# i18n Standards

## Setup

- Library: `next-intl`
- Locales: `ar` (Arabic, RTL, **default**) and `en` (English, LTR)
- Routing: all routes under `app/[locale]/...`
- Config: `i18n/routing.ts` (locale definitions) and `i18n/request.ts` (message loading)

## Translation Files

- `messages/ar.json` — Arabic (must always be complete)
- `messages/en.json` — English (supplementary)

Arabic is the default locale. Every key added to `ar.json` must also be added to `en.json`. The reverse is not enforced, but aim for parity.

## Translation Key Naming

Use dot-notation with a component or feature prefix (e.g. `"nav.home"`, `"quran.page"`). Keys should be descriptive, not generic. Avoid single-word keys at the top level.

## Using Translations in Server Components

```ts
import { getTranslations } from "next-intl/server";

export default async function MyPage({ params: { locale } }: ...) {
  setRequestLocale(locale); // Always call this first
  const t = await getTranslations("nav");
  return <h1>{t("home")}</h1>;
}
```

**Always call `setRequestLocale(locale)` before accessing translations in async server components.**

## Using Translations in Client Components

```ts
import { useTranslations } from "next-intl";

export function MyComponent() {
  const t = useTranslations("quran");
  return <span>{t("page")}</span>;
}
```

## Direction

Direction is set at the layout level. Use the helper from `app/utils/i18n.ts` to determine `dir`:

```ts
import { getDirection } from "@utils/i18n";
const dir = getDirection(locale); // "rtl" | "ltr"
```

For Quran text: always set `dir="rtl"` explicitly, even if the surrounding layout is LTR (e.g. English locale viewing Quran text).

## Routing Navigation

Use the locale-aware navigation from `i18n/routing.ts` instead of Next.js built-ins:

```ts
import { Link, redirect, usePathname, useRouter } from "@/i18n/routing";
```

## Extracting New Strings

Run `npm run extract-translations` to sync new translation keys from the codebase into the message files.
